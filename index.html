一、報告摘要與三方驗證

v2 方案（2026-04-09 交付）基於 LRP 原始碼 + 五崧保留清單（115 年度），設計「站碼字母 E / F / G / H / J + base-31 英數序號」作為流水號核心。v2 交付後 2026-04-21 收到貴公司提供的 2026 年度完整《報單號碼規則 Excel》，我們隨即展開三方交叉驗證（規則 Excel × LRP 原始碼 × DB 10 年實測 2.7 M 筆），發現 v2 的「站碼字母」是非必要的隔離機制——OP 實際上以 99.27–100% 的紀律遵守「Tier A 海運 pos10 = 收單關別第 1 字」規則。

本 v3.2 放棄站碼字母設計，改採按23 類別分成 Tier A（最嚴）/ Tier B（中）/ Tier C（寬）三層流水號規則，並用 Layer 1 autocomplete 把過去靠 OP 紀律的規則改成系統強制。32 條規則已逐一分類為：LRP 已實作、OP 紀律守到、或 LRP 未 follow，詳見附錄 B。

方案總結（v3.2）：把《2026 規則 Excel》的 23 個報單類別分為三層 Tier（A 最嚴 / B 中 / C 寬），每層各自決定流水號空間的 charset、pos10 規則與 A72 查重需求。放棄 v2 的站碼字母，改由 Layer 1 autocomplete 系統化強制 OP 過去靠紀律守到的規則。

三、三 Tier 動態流水號規則

《2026 報單號碼規則 Excel》明訂流水號（pos10-14）依報單類別分三層嚴格度。新系統按 declarationType 動態決定所屬 Tier，並配合 transportMode（海 / 空）決定 charset 與 pos10 策略。

3.1 Tier 分級

Tier A 最嚴 — 空運 NUMERIC 5 碼 / 海運 pos10 鎖收單首字 + 4 英數 + A72 查重

適用類別：F2 / F3 / D2 / D7（本體）+ B6 / D8（自由港版）

空運規則：pos10-14 = 純數字 5 碼（charset = NUMERIC，容量 99,999）

海運規則：pos10 = 收單關別第 1 字（例如 CW → C） + pos11-14 BASE31 4 碼

A72：跨類別查重 SUBSTR(DECL_NO, 5, 10)，僅限自由港子集

DB 吻合率：F3 99.87% / F2 99.59% / D2 99.27% / D7 99.46% / B6 FTZ 99.32% / D8 FTZ 100%（10 年 OP 紀律達標）

Tier B 中 — 海運 pos10 任意英字（實務偏 E/I） + 4 英數

適用類別：G1 / G3 / G5 / G7（一般）/ B6 / B8 / B9 / D8（一般）/ F1 / F5（一般）/ L1（物流中心）/ D5（一般）

空運規則：pos10-14 = 5 英數字（charset = BASE31，排除 I / L / O；DB 實測 100% 純數字）

海運規則：pos10 預設 E（出口）/ I（進口），OP 可改其他英字

DB 吻合率：G1 pos10='I' 72.5% / G5 'E' 71.4% / G3 'E' 88.5% / L1 'I' 85.7%（主流吻合）

Tier C 寬 — 海運 pos10 不鎖（預設 pos1 可改） + 4 英數

適用類別：T1（轉儲）/ F4（自由港區間）/ D1（課稅區/保稅倉）/ D5 / B8 / B9（自由港版）/ F5（船舶供油）

空運規則：pos10-14 = 5 英數字（charset = BASE31）

海運規則：pos10 預設 pos1（收單關別第 1 字）、OP 可改任意英數字

DB 吻合率：D1 pos1=pos10 84% / F4 60% / T1 偏純數字 79.7%（實務反而偏嚴）

3.4 全局黑名單

pos10-14 禁止字母（所有 Tier 通用）：I / L / O
原因：與數字 1 / 1 / 0 混淆，海關官方規定排除。LRP 原始碼 PUB_FUNC1.PAS L1274 有邏輯但被註解掉，新系統改 const 強制。
空運 pos10 禁止字母（全空運額外）：A / B / C / D
原因：避開關區代碼保留區。DB 實測洋基 / FedEx 經辦 45 萬筆空運零踩。全空運擋，不用識別快遞關區。

四、A72 跨類別查重

財政部關務署 114.6.9 修正的不受理原因代碼 A72：自由貿易港區報單（F2 / F3 / B6 FTZ / D8 FTZ）第 5-14 碼不可重複。LRP 原始碼 lcGet_4() 有實作此邏輯，新系統沿用並強化。

4.1 為什麼需要跨類別查重

A72 限制是 跨類別的：即便 F2 跟 B6 FTZ 是不同報單類別，若兩者的 pos5-14 相同仍會被海關拒收。LRP 的解法是：

// LRP/prodecl/PUB_FUNC.pas lcGet_4() SELECT COUNT(*) FROM SCUSDEC WHERE SUBSTR(DECL_NO, 5, 10) = '15012E0001' AND DECL_TYPE IN ('F2','F3','B6','D8') // Tier A 自由港子集 AND WAREHSE_YN = 'Y'; // 僅 FTZ 版 B6/D8

4.2 DB 實證（2021-2025）

驗證項目筆數結果
F2 / F3 / B6 FTZ / D8 FTZ 實際 pos5-14 重複組合 0 筆 LRP 實作有效
跨類別 pos5-14 相同但類別不同的「潛在衝突」組合 584+ 筆 都是 LRP lcGet_4() 查重後成功避開 → 證明保護機制作用中

4.3 新系統 checkCollision()

async function checkCollision( declType: DeclarationType, pos5to14: string, isFreeTradeZone: boolean, ): Promise<boolean> { if (!needsA72Check(declType, isFreeTradeZone)) return false; // 查自由港區子集內是否已存在相同 pos5-14 const exists = await this.dataSource .createQueryBuilder() .from(CustomsDeclaration, 'd') .where('SUBSTR(d.decl_no, 5, 10) = :key', { key: pos5to14 }) .andWhere('d.declaration_type IN (:...types)', { types: ['F2', 'F3', 'B6_FTZ', 'D8_FTZ'], }) .getCount(); return exists > 0; }

比 LRP 進步處：新系統把「判斷是否需查重」的邏輯封裝在 needsA72Check()，避免誤查（像 LRP 有時漏判 FTZ flag 導致漏查）。

五、監管編號處理（B1 / B2 / G2 特殊 4+4 切法）

B1（保稅廠課稅區售與）、B2（保稅廠相互交易）、G2（本地補稅）三個類別採特殊 4+4 切法：pos7-10 是監管編號後 4 碼 + pos11-14 是自編序號 4 碼（其他類別是 3+5 切法）。此規則在 2026 Excel 中明訂，但 LRP 原始碼的實作在「字母補位方向」上 B1/B2 與 G2 規則相反，是三方驗證發現的關鍵差異。

5.4 CustomerBondedBranch entity 職責

pos7-10 的監管編號來源是客戶主檔的保稅分支（一個客戶可能有多個保稅分支，每個分支 5 碼 bfno）。新系統獨立一個 entity CustomerBondedBranch（plan: 2026-04-21-customer-department-entity.md）來管理：

  • 不遷移 LRP 的 scusdec.td_bfno 欄位（DB 實測 B1 此欄位語意不穩：td_bfno='C6412' 但 pos7-10 可能是 6045 / 7040 / 5310 三組互不相關的值）
  • OP 打報單時，選擇買方 → 系統從 CustomerBondedBranch join 找對應 bfno → buildDeclNoPos7to10() 自動組字
  • 出口 / 進口方向不同（B1/B2 出口 vs G2 進口），自動對照正確補字方向

5.5 園區字母 vs 業者身份字母（2026-04-22 DB 深挖發現）

DB 實測發現 custdep.bfno 第 2 碼還有兩個字母沒被 Excel 列入園區字母：L（113 筆）和 D（87 筆）。深挖發現這兩個字母跟園區無關，是業者身份分類：

類型字母定義是否參與取號
園區字母 S / R / N / K / Q / T / C / V 科學園區 / 科技產業園區 / 農業生技園區 ✅ B1/B2 後補、G2 前補
業者身份字母⭐ 新發現 L 自由貿易港區物流中心業者（近鐵/瑞肯/郵船等） ❌ 純分類，不組字
業者身份字母⭐ 新發現 D 自由貿易港區內事業 / 製造業（帆宣/華立/立捷等） ❌ 純分類，不組字

八、新舊方案對比（v2 → v3.2）

v2（2026-04-09）主要設計核心是「站碼字母 E / F / G / H / J + base-31 英數序號」。v3.2 在規則 Excel + DB 三方驗證後發現 v2 的站碼字母是非必要的隔離機制，因此改為按類別分層的 Tier 策略。

❌ v2（2026-04-09）

站碼字母做為核心隔離機制

  • 海運 pos10 = 站碼字母（E 新竹 / F 桃園...）
  • 流水號 charset 全 BASE31
  • counter_key 不含類別
  • 保留區海 / 空運都有
  • A72 靠站碼字母天然隔離
  • 監管編號（B1/B2/G2）未處理

✅ v3.2（2026-04-22）

按類別分 Tier A/B/C 動態策略

  • 海運 pos10 按 Tier 決定（A 鎖收單首字 / B 預設 E/I / C 預設 pos1 可改）
  • Tier A 空運 NUMERIC 5 碼 / 其他 BASE31
  • counter_key 含 declType（避免跨類別衝突）
  • 保留區僅空運（海運靠 Service 層查重）
  • A72 用 checkCollision() 明確跨類別查重
  • buildDeclNoPos7to10() 處理 B1/B2 後補 vs G2 前補

8.1 為什麼放棄站碼字母

v2 站碼字母的核心訴求是「OP 需從流水號辨識站別」。但三方驗證發現：

理由證據
DB 實測 OP 已經用「pos10 = 收單關別第 1 字」做站別辨識 F3 99.87% / F2 99.59% / D2 99.27% / D7 99.46%（Tier A 海運各類別）
站碼字母跟收單關別第 1 字在資訊上重複（OP 看 pos1 = 關別 = 站別） 站碼只是 A/B/C/D（關別）→ E/F/G/H/J（站別）的硬 map
v2 要額外維護一張「站碼字母 → 辦公室」對照表 AJ100=E / AE100=F / AH100=G / AF110=H / AF100=J
v2 設計會佔用 pos10 一個字元，壓縮 Tier B/C 容量 BASE31 4 碼（含站碼）= 92 萬 vs BASE31 5 碼 = 2,863 萬

8.2 v3.2 的額外收穫

  • 覆蓋 23 類別全規則：v2 只做基礎取號，v3.2 含附錄 A 完整對照 + pos3-4 四種定義 + A72 跨類別查重
  • B1 / B2 / G2 監管編號自動化：v2 未處理，v3.2 透過 CustomerBondedBranch 自動帶入 + declType 分歧補字
  • OP 紀律規則系統化：I/L/O 黑名單、快遞 A/B/C/D 黑名單、Tier A 海運 pos10=收單首字，全部由 Layer 1 autocomplete 強制
  • 容量告警精細化：Task B 實測數據 2.5% 使用率，告警閾值從 v2 的「容量足」改為明確的 50 K / 80 K / 99 K 三級
